## ✅ Working

- **Free plan activation skips Midtrans cleanly** — `PaymentController::checkout` line 38-43 short-circuits to `activateFreePlan()` when `$plan->price <= 0`. Free plan promo discounts are explicitly ignored (line 39-41 comment). The free row is `forceCreate`d with `status='active'`, `payment_method='free'`, `paid_at=now()` — bypasses `$guarded`, no Snap token minted.
- **PaymentController::isEnabled gate is consistently applied** — `PaymentController.php:34, 192`, `GiftSubscriptionController.php:57, 80, 173`, `plans/index.blade.php:131`. Every Midtrans-using path checks before hitting `\Midtrans\Config`. "Coming Soon" UI is uniform.
- **Subscription model is mass-assignment hardened** — `$guarded` denylists `status`, `transaction_id`, `order_id`, `payment_method`, `paid_at`, `cancelled_at`, `amount`. Every write goes through `forceCreate`/`forceFill` from server-trusted call sites. `billing_address` is encrypted at rest (line 43).
- **Promo validate/apply split is well-designed** — `validateCode()` is pure read-only, called from `validatePromo` JSON endpoint + the round-trip `checkout` action. `apply()` is invoked only by the webhook after Midtrans confirms settlement, wrapped in `DB::transaction` with `lockForUpdate` on the parent code row and re-validates inside the transaction so a code that hit `max_uses` between checkout and webhook is rejected without writing a phantom redemption.
- **Promo redemption ledger is atomic + idempotent** — unique `(user_id, promo_code_id, subscription_id)` lookup at `PromoCodeService::apply()` line 102-109 returns the existing discount on retry instead of writing twice. Webhook retries cannot double-burn `max_uses`.
- **Promo `isRedeemableBy` covers every gate** — active flag, `[starts_at, expires_at]` window, global `max_uses`, per-user cap, `applies_to_plans` whitelist (null = any), `is_first_time_only` (paid past subs only — free plans excluded so seeding the Free plan doesn't burn WELCOME10).
- **Promo case-insensitivity is collation-independent** — `PromoCode::setCodeAttribute` always uppercases on write (`PromoCode.php:83-86`), `PromoCodeService::validateCode` uppercases on read (`PromoCodeService.php:39`). Works on every DB collation without depending on `utf8mb4_ci`.
- **Promo discount line item is transparent** — `PaymentController::checkout` line 87-94 emits the discount as a separate negative item in the Midtrans payload so the customer's receipt shows "PROMO: -Rp 5.000" rather than just a reduced gross_amount. The `custom_field1` carries the code text for Midtrans-dashboard search.
- **Promo pending-id is cached by order_id** — line 139-145 stashes the chosen `promo_code_id` in `cache()` keyed by `order_id` (24h TTL). Webhook pulls it line 230-240. Means the promo cannot be lost to a user opening checkout in two tabs and changing codes — the order_id is the unique key, not the user.
- **Gift purchase + redeem is racing-safe** — `GiftSubscriptionService::redeem` line 116-174 wraps the whole flow in `DB::transaction` with `lockForUpdate` on the gift row, then double-checks `redeemed_by_user_id` not set by someone else. Lifecycle (`isPaid`, `isRedeemed`, `isExpired`) gates correctly. Concurrent redeem clicks return one Subscription, not two.
- **Gift code generation is collision-safe + ambiguity-safe** — `GiftSubscription::makeUniqueCode` excludes `0/O/1/I/L` glyphs (line 130), 8 retries before fallback `Str::random(16)`. Unique index on `code` is the actual backstop.
- **Gift redeem accepts both raw and human-friendly codes** — `GiftSubscriptionService::redeem` line 110 strips `GIFT-`, dashes, spaces before lookup. User can paste either `ABCDEFGHJKMN` or `GIFT-ABCD-EFGH-JKMN`.
- **Gift webhook is idempotent + email-fires-once** — `GiftSubscriptionController::webhook` line 207 captures `firstTime = $gift->paid_at === null`, only sends `sendGiftEmail` on first arrival. Email failure is logged but never blocks the webhook ack (line 217-223).
- **Gift webhook forwards non-gift orders** — line 189-191: if `order_id` doesn't start with `GIFT-`, defers to `PaymentController::webhook`. Ops can point Midtrans at a single URL.
- **Referral attribution is race-safe** — `ReferralService::attribute` returns the existing conversion if `referred_by_user_id` is already set (line 71-77). Self-referral guard at line 86-88. Unique index `(referrer_user_id, referred_user_id)` on `referral_conversions` is the hard backstop. Wrapped in `DB::transaction`.
- **Referral reward idempotency** — `markRewarded` checks `status === REWARDED` and bails (line 197-199). Webhook retries don't double-pay 500 coins. Status backfill from `pending` → `rewarded` (skipping `qualified`) is handled cleanly (line 202-213).
- **Referral counter bump uses SQL atomic increment** — `ReferralService.php:240-241` uses `DB::raw('referrals_count + 1')` so concurrent rewards never lose increments.
- **/r/{code} capture is well-bounded** — pattern constraint `[A-Za-z0-9]{6,16}` at `routes/web.php:407` rejects garbage at routing layer; ReferralController re-validates with `preg_match` line 102; throttle 30/min/IP; cookie is HTTP-only, SameSite=lax, secure when request is secure, 30-day TTL.
- **Promo bulk-generate has loop safety** — `PromoCodeController::bulkGenerate` line 144 caps attempts at `count * 4` so a near-collision template can't infinite-loop.
- **PromoCodeReport conversion-uplift metric is correctly framed** — `PromoCodeReportController.php:74-88` joins distinct subscription_ids from the redemption ledger against `Subscription` rows with `amount > 0` over a 30-day window — that's the right denominator.

## 🔧 Needs improvement

- 🔴 **Midtrans webhook signature is NEVER verified explicitly.** Both `PaymentController::webhook` and `GiftSubscriptionController::webhook` instantiate `new \Midtrans\Notification()` and rely entirely on the SDK to validate the `signature_key` header against `order_id + status_code + gross_amount + server_key`. There is no application-layer signature check, no IP allowlist, no replay-window check. A typo or library swap (e.g. mocking `\Midtrans\Notification` for tests) would silently disable signature verification. **At minimum**: add `abort_unless($notification->isVerified(), 401);` immediately after instantiation (the SDK exposes `isVerified()`), and assert the `Authorization`/`Signature` header is present BEFORE instantiating the SDK so a webhook without a header is rejected cheaply. Also: pin the webhook URL behind an `allowed_ips` middleware sourced from Midtrans' published callback IPs.
- 🔴 **`order_id` is built from `time()`, not from a cryptographic source.** `PaymentController.php:46`: `'FLIK-' . $user->id . '-' . time()`. Two simultaneous checkouts in the same wall-clock second collide → second one fails Midtrans uniqueness. Same `'GIFT-'.$gift->id.'-'.time()` at line 112. Replace with `Str::ulid()` or `Str::uuid()`.
- 🔴 **`SubscriptionPlan::$fillable` is missing `duration_days`** (line 12-16). The column exists (added in `2026_03_08_000001_*`) and is read by `PaymentController` line 81, 130, 380 — but admin CRUD using `$plan->update($request->validated())` cannot persist it. Today plans are seeded server-side so no symptom; the moment someone tries to edit a plan with a custom duration through any future admin form, the value will silently drop.
- 🔴 **`SubscriptionPlan::price` cast is `decimal:2` but checkout casts to `(int)` everywhere.** `PaymentController.php:51, 79, 99, 130`. Any plan with a non-integer rupiah price (none today, but the column allows it) will be silently truncated. Either move the column to `unsignedInteger` or drop the `(int)` casts and pass the decimal through to Midtrans.
- 🔴 **`PaymentController::webhook` does not re-check `subscription->status` before activating.** If a subscription was manually cancelled by admin (status='cancelled'), a delayed Midtrans webhook (e.g. expire → settlement race) will flip it back to 'active' and add 30 days. Guard with `if ($subscription->status === 'cancelled') return ack;`.
- 🔴 **`PaymentController::activateFreePlan` has no anti-abuse check.** A user can hit `/checkout/{free_plan}` repeatedly and accumulate dozens of overlapping free subscriptions (one row per click). At minimum require `!user->subscriptions()->active()->exists()`. Also the user data export / refund report will be polluted.
- 🔴 **Subscription `ends_at` is calculated client-side trusting `$plan->duration_days`** (line 130, 380). Promo type `free_trial_days` is documented to "add the actual extra days to ends_at by the controller" (`PromoCode.php:233-234`) but **no such code exists** in PaymentController. The promo type is essentially dead — it just zero-bills the customer and grants the regular `duration_days`, not the extra trial days.
- 🔴 **Refund flow does not exist.** Confirmed by grep across `app/`. The Midtrans `refund` / `refund/online/direct` API endpoints are not wrapped. Admin must issue refunds on the Midtrans dashboard side and then manually toggle `subscriptions.status='cancelled'` via tinker. The `PromoCodeRedemption` ledger row stays attached to a refunded txn, polluting marketing reports. The `ReferralConversion.status='rewarded'` row also stays — and the coins were already granted to the referrer — opening a refund-farming vector (signup → pay → refund → referrer keeps 500 coins).
- 🔴 **Referral reward farming vector.** `markRewarded` is called on `settlement` but never reversed on `cancel`/`refund`/`chargeback`. A referrer earns 500 coins per friend they convince to pay one month and immediately refund. Add a `ReferralService::reverse(User $referred)` and call it from the `deny`/`cancel`/`expire` and `chargeback` branches of the webhook.
- 🔴 **`PaymentController::webhook` is missing the `'success'` cancel branch for `cancel` AFTER `settlement`.** Midtrans can fire `cancel` on a refund. The webhook would re-enter the `in_array($status, ['deny', 'cancel', 'expire'])` branch and flip `status='cancelled'` — but the user's 30-day grant still stands (`ends_at` not touched). User keeps Premium until ends_at while admin sees status='cancelled'.
- 🔴 **`PaymentController::webhook` JSON 500 on exception breaks Midtrans idempotency.** Line 320-322 returns 500 with the error message in the body. Midtrans treats 5xx as "retry forever". A bug in the redemption code that throws (the `try/catch` around `PromoCodeService::apply` is good, but the outer `try/catch` around the whole webhook is what catches the bug) will cause infinite retries that re-process payment+xp+notifications. Always 200 the webhook once the subscription update succeeded; downgrade errors to log+ack.
- 🔴 **Webhook awards XP unconditionally on every settlement** — `PaymentController.php:259-260`. Re-delivered webhooks (Midtrans retries until 200) before the unique-redemption guard kicks in will double-award. Wrap the XP grant in the same "first transition to active" guard as the `Notification::create` should be (`$wasPending = $subscription->status === 'pending'`).
- 🔴 **Promo `discount_value` for percentage type allows >100** — clamped only inside `computeDiscount` (line 230 `min(100, max(0, ...))`) but `validatePayload` allows any `numeric|min:0`. Admin can enter `200%`, validator passes, label renders "200%" but discount caps at base price. Either reject in validation (`max:100` when type=percentage) or warn in UI.
- 🔴 **Promo `applies_to_plans` array stores plan IDs in JSON but no FK integrity** — if a plan is hard-deleted, the JSON entry dangles silently. Migration uses `nullOnDelete` on `created_by_user_id` but JSON has no FK guarantee. On plan delete, run a sweep to strip the ID from every promo.
- 🔴 **Gift purchase auth-optional but expires_at is hard-coded to 90 days at model boot** — `GiftSubscription::booted` line 53-55. Buyer cannot choose duration, admin cannot extend. For corporate-gifting flows (the obvious B2B use case) this is too rigid.
- 🟡 **Snap script tag mounts even when `payment.checkout` is unreachable.** `payment/checkout.blade.php:100` injects `https://app.{is_production ? '' : 'sandbox.'}midtrans.com/snap/snap.js`. If Midtrans is down, the page renders but `snap.pay()` is undefined — clicking Pay does nothing visible. Add `<script>` `onerror` handler that flips the button to "Coming Soon".
- 🟡 **Promo Apply button does a full page reload via `window.location.href`** (`payment/checkout.blade.php:179`). The Snap token is regenerated but the user's typed promo string is preserved in the URL `?promo_code=…`. That URL leaks the code to the Referer header on any outbound link click from the checkout page — not catastrophic, but a paranoid user complaining about "my promo code is in my history" is plausible.
- 🟡 **`PromoCodeService::apply` re-validates inside the transaction but the controller never refunds the customer on race rejection.** If a promo hits `max_uses` between checkout and webhook, the customer is charged the FULL (non-discounted) gross_amount via Midtrans (because the Snap token was already minted at the discounted amount — the gateway's gross_amount IS what gets charged). The service logs and bails, but the user paid the discounted amount and got no redemption row, which is actually fine, but admin can never tell from the ledger that the discount-was-honoured-but-redemption-was-not. Persist a "skipped" reason somewhere.
- 🟡 **`Notification::create` on payment success uses emoji in `title` (`'🎉 Langganan Aktif!'`)** — fine, but the project's CLAUDE.md note "only emojis when user explicitly requests it" suggests the convention should be drier copy. Same in `'🎁 Hadiah berhasil ditukar!'`, `'🎉 Referralmu Aktif!'`, `'💰 Bonus Referral Besar!'`.
- 🟡 **`PaymentController::webhook` writes the success notification BEFORE the promo apply runs.** If a malicious promo manipulation somehow gets through but the notification is already sent, the user sees "subscription active!" + a customer-support ticket because their discount didn't apply. Order: activate sub → apply promo → fire notification.
- 🟡 **Gift cookie referral capture and Snap checkout cookie collide on subdomain.** Gift purchase is auth-optional and writes the Midtrans checkout cookies; if the buyer ALSO has a `flik_ref` cookie from clicking `/r/CODE` earlier, the post-purchase flow does not "convert" them (they're not signing up). - 🟡 **Subscription `cancelled_at` is never written.** `PaymentController.php:289` flips `status='cancelled'` but does not stamp `cancelled_at` (the column exists in the original migration). Admin reports group cancellations by `updated_at` instead, losing fidelity.

The `intended()` call requires `Auth::guest()` to have been set; an already-authed user hits `/r/CODE` and lands on `/movies` losing their session intent. ## 📊 Satisfaction Score: 3/10

**Justification**: The architecture is excellent on paper — promo validate/apply split with transaction + lock + re-validation, idempotent gift redemption, race-safe referral attribution with unique constraint, denormalised counters with SQL atomic increments, encrypted billing_address, mass-assignment hardened Subscription, ENV-gated `isEnabled()` mirrored across every Midtrans-using path. The schema design is genuinely thoughtful (the `(is_active, expires_at)` composite index, append-only redemption ledger with frozen `discount_applied_idr`, the `(referrer, referred)` unique pair). The service-layer abstraction (`PromoCodeService`, `GiftSubscriptionService`, `ReferralService`) is exactly the right shape and is consistently invoked from controllers.

But the actual deliverables are catastrophically incomplete:

1. **Midtrans SDK is not installed.** Every checkout 500s today. This is the headline issue and it dwarfs everything else.

## Bonus: missing-but-expected features

- **Recurring billing / auto-renew.** Every Subscription is a fixed-window grant. Real OTT (Netflix, Disney+) auto-renews. Midtrans Snap supports tokenisation for recurring CC — the code has zero scaffolding for it.
- **Proration on plan upgrade/downgrade.** User on Basic upgrading to Premium mid-cycle should get a prorated charge; today they pay full Premium and lose the unused Basic days.
- **Free trial.** The `free_trial_days` promo type exists but the controller never adds days to `ends_at` — there is no real trial offering, just a marketing-flavoured zero-price code.
- **Apple Pay / Google Pay** — Midtrans Snap supports them natively but the integration is generic.
- **In-app subscription management page.** Users can subscribe but have no UI to view their current plan, billing history, next renewal date, or cancel. `/profile` shows a flash-only success message.
- **Cancel/pause subscription self-service.** `Subscription::scopeActive` filters by `status='active'` but no controller exposes a "cancel" action. Admin must do it via tinker.
- **Annual plan UI** — `plans/index.blade.php` toggles monthly/yearly with a hardcoded 20% discount in JS, but the actual `subscription_plans.billing_cycle='yearly'` rows do not exist; clicking the yearly Premium button still hits the monthly plan's checkout URL.
- **Receipt / invoice download** — no PDF generation, no email after payment carrying an invoice. Required for B2B and tax purposes in Indonesia.
- **VAT/PPN handling** — Indonesia requires 11% PPN on digital services from a certain revenue threshold. Zero handling.
- **Tiered referral rewards** — currently flat 500 coins per first conversion. Real programs ladder (5 friends = 1000 bonus, 10 = silver tier, etc.).
- **Gift card balances / store credit** — gifts here are full-subscription unlocks, not flexible store credit. Users can't accumulate balance.
- **Promo stacking / combinable codes** — only one code per checkout; no stacking, no combinable codes.
- **Promo via direct URL** — `/checkout/{plan}?promo_code=WELCOME10` works, but there's no shareable `/promo/{code}` landing page that auto-applies to whatever plan the user picks.
- **Webhook deduplication via `transaction_id`** — should write each Midtrans `transaction_id` to a `payment_webhook_log` table on receipt + uniquely index it, returning 200 if already seen. Today retries hit the subscription row repeatedly.
- **Audit log entries for payment events.** `AuditLogger` is referenced in CLAUDE.md but no payment path writes to it. Failed signature checks, refunds, manual cancellations should all be auditable.
- **Coupon-eligibility preview on plans page** — user enters a code on `/plans` and sees which plans it applies to before clicking checkout.
- **Dunning / failed-payment retry email cadence.**
- **Geographic pricing** — single IDR price for everyone; no per-country adjustment despite the `GeoIpResolver` being present in the stack.
- **Subscription gifting from within the app** — `/gift/{plan}` works for net-new buyers but no in-app "send this to a friend" CTA on `/plans` or `/profile`.
- **Pre-paid bundles** (3-pack of 1-month codes at a discount, school-pricing) — common in Indonesia (Spotify, Netflix education partnerships) and absent.
