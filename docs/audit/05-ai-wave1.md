## ✅ Working

- **Provider switching works end-to-end.** `AiClient::pickProvider()` calls `AiProvider::default()` on every `chat()` call (`app/Services/Ai/AiClient.php:44,176`); `AiProvider::default()` (`app/Models/AiProvider.php:122`) returns `is_active=true AND is_default=true` (fallback: first active by priority). `AdminController::storeAiProvider/updateAiProvider` (`AdminController.php:568-570, 599-603`) demote any other default row before setting the new one. There is no caching of the picked provider, so flipping default in `/admin/ai-settings` re-routes the very next call. **Every Wave 1 task that goes through `AiClient` will follow the switch.**
- **`UsageTracker` runs on every `AiClient::chat()` call — success and failure.** `AiClient.php:119-131` writes on success (with token counts + cost); `AiClient.php:151-166` writes on failure (tokens=0, `success=false`, `error_message` truncated to 2000 chars). Tracker wraps its own writes in try/catch so a logging failure cannot break the calling code. `forceCreate` is used because `AiUsageLog::$guarded = ['*']` (intentional system-only write surface — `app/Models/AiUsageLog.php:22`).
- **Pricing table covers 26 named models** including the current default `claude-opus-4-7` at $15/$75 per MTok (`UsageTracker.php:43`). Fallback `[1.00, 3.00]` per MTok keeps unknown models from logging $0.
- **`/admin/ai-usage` dashboard is fully populated by real data.** Stats query (`AiUsageController.php:22-51`) returns spend 7d/30d, success rate, avg latency, tokens; per-provider LEFT-JOIN includes zero-usage providers; per-task `GROUP BY task_type`; 14-day sparkline zero-fills missing days (line 119-129). All read from `ai_usage_logs`.
- **`/admin/ai-settings` Test Connection** — `ProviderTester::test` (line 49) re-implements the request shape per provider type (anthropic/google/openai-compat), runs a 16-token `"Reply with: OK"` probe with a 15s timeout, returns `{success, latency_ms, response, error, model, usage}` JSON for the Alpine.js inline widget (`ai-settings.blade.php:69+`). Crucially does **not** mutate `last_used_at` or token counts — it's a diagnostic, not real usage. Honors SSRF guard (line 89-93).
- **SSRF protection on every outbound provider call.** `AiClient::chat` (line 76) and `ProviderTester::test` (line 90) both call `SsrfGuard::assertUrlAllowed($endpoint)` before the HTTP request. `WebSearchService` routes Wikipedia + DuckDuckGo through `SafeHttp` (line 19) which delegates to the same guard. `ThumbnailPicker::scoreFrame` line 207 also calls `(new SsrfGuard())->assertUrlAllowed(...)`. Mitigates a malicious admin pasting `http://169.254.169.254/...` as `base_url`.
- **Auto-moderate + auto-spoiler-detect on new comments** — `EventServiceProvider::boot` (lines 69-90) subscribes to `Comment::created` and dispatches `ModerateComment` + `DetectSpoilerInComment` jobs (both `ai-realtime`). Both jobs skip rows already processed (`moderated_at` / `spoiler_checked_at` not null) and fail-open: on AI exception, `CommentModerator` marks status `pending` and keeps `is_visible=true`; `SpoilerDetector` keeps the user's self-flag.
- **`MovieReviewer` is idempotent** via `updateOrCreate` on `(movie_id, perspective)` and gracefully clamps title to 180 chars (`MovieReviewer.php:209-212`). Falls back to a label-derived title when the model ignores the strict-format contract.
- **`TriviaGenerator` is grounded in Wikipedia** via `WebSearchService::search(... + ' film')` and prompted to verify against the snippet (line 70-77). Persists `source_url` from the first hit so admins can audit. Idempotent (delete-then-batch-insert in a transaction, line 134-137).
- **`QuoteExtractor` sampling** caps prompt size at 2000 cues with uniform-step sampling (line 162-169) — long features don't blow the token budget.
- **`SpoilerDetector` respects user self-flag below confidence threshold** (line 110-113): only overrides `is_spoiler` when AI confidence ≥0.7; below that, OR's the AI verdict with the user flag (so an admitted spoiler stays flagged).
- **`CommentSentimentAnalyzer` bulk mode chunks at 40 comments per call** (`BULK_CHUNK_SIZE`) keeping any single failure bounded, and accepts out-of-order responses via the `index` field.
- **`MovieTagger` strict-JSON validation** — coerces `era` to `YYYYs`, clamps each string list, restricts `intensity` to `{low,medium,high}`. Treats all-empty output as failure (line 153-158).
- **`SeoMetaGenerator` defensive truncation** to DB column lengths (60/160/500 chars) regardless of model output, with sensible Indonesian fallbacks (`fallbackTitle/Description/Keywords` lines 156-184).
- **`PromoBannerGenerator` / `SocialMediaPostGenerator`** — multibyte-safe length clamps, sensible fallbacks (movie title + overview, default hashtags from genres), Twitter math correctly reserves space for inline hashtags (`SocialMediaPostGenerator.php:151-170`). Hashtag normaliser strips spaces, dedupes via array key, caps at 12.
- **`DailyAdminReportGenerator`** — well-engineered aggregator with per-plan revenue breakdown, MAU/DAU/watch-hours, top movies, keyword-based comment sentiment classifier (no AI tokens spent for that step), AI narrative call with explicit Indonesian system+user prompts, **and** a deterministic `fallbackNarrative` if the AI call fails (line 298-316). Scheduled daily 07:00 → super_admin emails (`SendDailyAdminReport.php`, `app/Console/Kernel.php:30`).
- **`EmailPersonalizer` degrades gracefully** when `$ai` is null — returns deterministic Indonesian template copy per intent (`fallbackSubject/Body` lines 280-341). Profile builder eager-loads `WatchHistory.movie.genres` to avoid N+1.
- **Encrypted API keys on `AiProvider`** — `protected $casts = ['api_key' => 'encrypted']` + `$hidden = ['api_key']`. Masked-key accessor (`getMaskedApiKeyAttribute`) shows first 6 + last 4 only and is hardened with try/catch so a key rotation doesn't 500 the admin page (`AiProvider.php:108-119`).
- **Job queue routing matches CLAUDE.md convention.** `ai-realtime` (`TagMovie`, `GenerateMovieSynopsis`, `GenerateMovieShortSummary`, `PickMovieThumbnail`, `ModerateComment`, `DetectSpoilerInComment`) vs `ai-batch` (`GenerateMovieSeo`, `GenerateMovieTrivia`, `ExtractMovieQuotes`, `SuggestTrailerCuts`, `AnalyzeCommentSentiment`). Backoff arrays look reasonable (e.g. `GenerateMovieSeo::backoff() = [30, 120, 600]`).
- **RBAC permissions for AI surface** seeded properly (`database/seeders/RolePermissionSeeder.php:48-50, 53-54, 62`): `ai.providers.configure`, `ai.usage.view`, `ai.tasks.run`, `marketing.banner`, `marketing.social`, `comments.moderate`. Every admin route uses `->middleware('can:...')` instead of relying solely on the `admin` gate.

## ⚠️ Inactive / not wired

The movie detail page (`resources/views/components/movies/show.blade.php`) embeds `ai-summary` / `ai-trivia-strip` / `ai-quotes` but never the SEO component. Net effect: **every AI-generated SEO meta tag is invisible to search engines** — the seven seconds of LLM time per movie produces zero SEO uplift. Fix: add `@push('head') <x-movie-seo :movie="$movieModel" /> @endpush` to `show.blade.php`.
- 🔴 **`AnalyzeCommentSentiment` job is dispatched from nowhere.** `grep -rn "AnalyzeCommentSentiment::dispatch\|new AnalyzeCommentSentiment" app/` returns zero hits. The `Comment::created` hooks in `EventServiceProvider::boot` (lines 69-105) dispatch `ModerateComment` and `DetectSpoilerInComment` but not sentiment. There is no `flik:sentiment:analyze` console command either. Net effect: **the `/admin/sentiment` dashboard is permanently empty** unless someone manually `app(CommentSentimentAnalyzer::class)->analyzeBulk(...)`s from tinker. The dashboard's controller (`SentimentDashboardController::overallStats`) even computes a `pending` count for un-analysed comments — that counter will keep climbing forever.
- 🔴 **Comment moderation hides nothing on the public page.** `CommentModerator` correctly sets `is_visible=false` and `moderation_status='flagged'` when AI confidence > 0.7 (`CommentModerator.php:96-106`), but the public render path `VelflixController::show` builds the comments query with **no `where('is_visible', true)` and no `whereIn('moderation_status', ['approved','pending'])` filter** (`VelflixController.php:339-349`). The migration's whole point — gating visibility on AI verdict — is bypassed. Even rejected admin-moderated comments stay visible to users. Fix: add `->where('is_visible', true)` to the public comments query (and to `Comment::topLevel()` scope if other callers need the same behaviour).
- 🟡 **`AnalyzeCommentSentiment` job ships with no scheduler entry** even though the analyzer's bulk mode is purpose-built for nightly batch runs. Add either a `Comment::created` dispatcher (mirroring the moderation/spoiler listeners) or a `flik:sentiment:backfill` daily command.
- 🟡 **`MovieReviewer` has no bulk command.** Reviews are admin-button-only via `/admin/movies/{movie}/ai-reviews/generate` (one perspective per click, sync — bounded only by `AiClient`'s 30s HTTP timeout). Catalog-wide review generation requires N×4 manual clicks. Add a `flik:ai:reviews-all --perspective=critic` command.
- 🟡 **`flik:ai:seo-all` is not scheduled.** Synopsis (`flik:ai:tag-all` Sat 01:00), bio (`flik:cast:enrich-bios` Sat 02:00), and intro-outro (`flik:detect:intro-outro` Sat 03:00) all run weekly; the SEO command (`flik:ai:seo-all`) — arguably the highest-ROI of the bunch — has no schedule entry. New movies remain SEO-blank until somebody runs it by hand. Add `$schedule->command('flik:ai:seo-all --limit=20')->dailyAt('04:00');` or similar.
- 🟡 **No bulk command for `MovieSummaryGenerator`'s short summary.** `AiGenerateShortSummary.php` exists with signature `ai:generate-short-summary` (not the `flik:` prefix used elsewhere) — confirm it actually runs. Naming inconsistency: `ai:pick-thumbnails`, `ai:suggest-trailers`, `ai:extract-quotes`, `ai:generate-trivia`, `ai:generate-short-summary` all use the legacy un-prefixed name; `flik:ai:tag-all`, `flik:ai:synopsis-all`, `flik:ai:seo-all`, `flik:ai:bts`, `flik:ai:weekly-digest`, `flik:ai:cinematography`, `flik:ai:highlights`, `flik:ai:quiz`, `flik:ai:spoilers` use the new `flik:` prefix. Every thumbnail-picking run is invisible on `/admin/ai-usage`. Either route through `AiClient::chat()` with a vision message shape (it currently only supports text — would need extending) or call `UsageTracker::track()` directly with `taskType: 'vision.thumbnail_pick'`.
- 🟡 **Only 3 of 16 Wave-1 tasks pass a non-default `taskType:` to `AiClient::chat()`** — namely `SpoilerDetector` (`comment.spoiler_detect`), `CinematographyAnalyzer` (`cinematography.narrative`), and `QuizQuestionGenerator` (`quiz.generate_questions`) per `grep -n "taskType:" app/Services/Ai/Tasks/*.php`. Every other task (`MovieTagger`, `SynopsisGenerator`, `MovieSummaryGenerator`, `QuoteExtractor`, `TriviaGenerator`, `MovieReviewer`, `CommentModerator`, `CommentSentimentAnalyzer`, `SeoMetaGenerator`, `PromoBannerGenerator`, `SocialMediaPostGenerator`, `EmailPersonalizer`, `DailyAdminReportGenerator`, `TrailerSuggester`) logs under the default label `'chat.generic'`. The per-task breakdown on `/admin/ai-usage` (`AiUsageController.php:85-98`) is therefore a single fat `'chat.generic'` row plus three small ones — useless for cost-attribution. Each task should pass an explicit label like `'tag.movie'`, `'synopsis.generate'`, `'review.critic'`, etc.
- 🟡 **Most tasks don't pass `$subject:` either.** `UsageTracker::track` accepts an optional `Model $subject` and writes it to `subject_type/subject_id` polymorphic columns, but every Wave-1 task call site (except `SpoilerDetector`) omits the argument. Per-movie spend analysis is therefore not possible from `ai_usage_logs` alone — you'd have to join through created_at ranges. Quick fix: add `subject: $movie` (or `$comment`, `$user`) to every `$this->ai->chat(...)` call.
- 🟡 **`AnalyzeCommentSentiment` queue routing inverts the operative class's intent.** Job hardcodes `$this->onQueue('ai-batch')` (line 43), but the single-comment path is invoked from end-user comment submission and benefits more from `ai-realtime` (the same queue moderation rides) so admins see the sentiment column refresh promptly. Either dispatch from `Comment::created` on `ai-realtime` for single mode and keep `ai-batch` for backfill bulk only, or split into two jobs.
- 🟡 **`flik:report:daily` schedules at 07:00 server time** (`app/Console/Kernel.php:30`) — fine for one timezone, but the report covers "yesterday 00:00–23:59" in **server local time**, not `Asia/Jakarta`. If the server runs UTC, "yesterday" ends at 07:00 WIB → super-admins get a report that's still missing the last 7 hours of the calendar day they think they're reading about. Either pin `Carbon::now('Asia/Jakarta')` in `DailyAdminReportGenerator::generate` or run the schedule at a `->timezone('Asia/Jakarta')` qualifier.
- 🟡 **Standalone helpers don't reach the contracts.** `AiClientContract` is bound (`AppServiceProvider.php:36`) but every task service type-hints the concrete `AiClient` class (`grep "protected AiClient" → 49 hits`). `KnowledgeSourceContract` and `WebSearchContract` aren't bound at all (no `app()->bind` for them anywhere). The "future microservices swap" the contracts were built for is impossible without re-typing every constructor. Either bind the contracts in `AppServiceProvider::register` AND re-type all tasks to the interfaces, or drop the contract interfaces — keeping them as decorative noise is a maintenance trap.
- 🟢 **`MovieReviewer.PROFILES['family'].max_tokens=450` for a 120-word brief.** All four profiles allocate ~3-4 tokens per Indonesian word, but the 200-word `critic` profile only gets 700 tokens (3.5x) while the 250-word `academic` profile gets 850 (3.4x). The maths is right; the warning is that Indonesian sometimes runs longer than ratio suggests — keep an eye on `finish_reason='length'` rates in the usage log.

## 🔧 Needs improvement

- 🔴 **`AiClient` never enforces a per-task max_tokens ceiling.** Callers pass `max_tokens` in `$options['max_tokens']` but `AiClient` happily forwards 100k if asked. A buggy task service or compromised admin who tweaked a prompt to "Generate 100,000-word review" would burn through the budget. Add a hard ceiling (e.g. 4096) at `AiClient.php:49` with a log warning when truncated.
- 🔴 **No retry / circuit-breaker around the provider HTTP call.** `AiClient::chat` does a single `Http::post` (line 101) with no retry — any transient 5xx fails the whole task, and `MovieTagger`/`TrailerSuggester`/etc. just log + move on. For paid providers (DeepSeek, OpenAI) where token spend is already booked client-side, one retry on 5xx + `Retry-After` honouring would be cheap insurance. Use `Http::retry(2, 250)`.
- 🔴 **`AiClient::pickProvider` queries `AiProvider::default()` on every call** — no in-request memoisation. The `DailyAdminReportGenerator` makes 1 call, but `CommentSentimentAnalyzer::analyzeBulk` chunks at 40 and could make hundreds of calls per request; each one hits the DB to re-resolve the default provider. Cache the picked provider on the `AiClient` instance for the lifetime of the request.
- 🔴 **`FilmKnowledgeService::findClosestByTitle` does `Movie::all(['id','slug','title'])` for token-overlap fallback** (line 208). With a 350-film catalog this is fine, but the comment in `CLAUDE.md` hints at growth — once the catalog reaches a few thousand films this is a per-RAG-call full table scan in PHP. Add a fulltext index migration or use the existing `slug` index plus a `LIMIT 50 ORDER BY popularity DESC` pre-filter.
- 🔴 **`FilmKnowledgeService::findByTitle` builds a SQL injection-shaped `LIKE`** (line 116). The value is `LIKE "%{$title}%"` with no sanitisation — `Movie::where('title', 'LIKE', $value)` is bind-safe, but `%` and `_` characters in the user input become wildcards the caller didn't intend. If a user searches for a film with `_` in the title (e.g. `Star_Wars`), it accidentally matches more than intended. Escape with `addcslashes($title, '%_\\')`.
- 🔴 **`MovieTagger::cleanStringList` lowercases everything, dropping case info** that downstream UI might want (e.g. for proper nouns in `audience`). Acceptable for this taxonomy but the function has no docblock warning; if someone reuses it for a freer field later they'll be surprised.
- 🔴 **`TriviaGenerator` and `QuoteExtractor` use `MovieTrivia::insert([...])` / `MovieQuote::create([...])`** without writing audit-log rows even though they're persisting AI-generated content. `AuditLogger::ai(...)` exists in this codebase but is unused here — admins can't see "who/what regenerated trivia for movie X at 14:32". Wire `app(\App\Services\Audit\AuditLogger::class)->info('ai.trivia.generated', subject: $movie, meta: ['count' => count($rows)])`.
- 🔴 **`CommentModerator::moderate` writes back via `$comment->forceFill(...)->save()`** (line 75-81 on failure, line 100-106 on success). `->save()` fires Eloquent model events including `Comment::updated`, which can fan out to observers and (because the `Comment::created` listener is in `boot()`) potentially re-trigger the moderation listener if anyone adds an `updated` hook later. Use the same pattern as `CommentSentimentAnalyzer::persist` — `Comment::query()->whereKey(...)->update([...])` to skip events.
- 🔴 **`SpoilerDetector::persistResult` calls `$comment->forceFill(...)->save()`** with the same fan-out risk. Same fix.
- 🔴 **`DailyAdminReportGenerator::commentsSummary` hand-rolls keyword sentiment** with hard-coded ID/EN word lists (line 220-221) instead of reading the freshly-populated `comments.sentiment` column the `CommentSentimentAnalyzer` writes. Result: the daily report sentiment is a different number from the dashboard sentiment, both claim to be authoritative, and the AI tokens spent on `CommentSentimentAnalyzer` never inform the executive narrative. Query `comments.sentiment` instead.
- 🔴 **`ProviderTester::test` recomputes ~120 lines of duplicate-of-`AiClient` payload/header/parse logic.** Provider-specific code lives in two places now — every new provider type needs to be added to both `AiClient.php` and `ProviderTester.php`. Extract the per-provider shape (buildPayload / buildHeaders / parseResponse / endpoint) into a `ProviderShape` value object and reuse.
- 🔴 **`AiClient` mutates `provider->update([...])` on every call** (`UsageTracker.php:122-125`) — two writes per task (one row insert + one provider row update). Under burst load (e.g. bulk sentiment job chunking at 40) you get N+1 writes to the same row, all racing. Use `AiProvider::query()->whereKey($provider->id)->update(['last_used_at' => now(), 'total_tokens_used' => DB::raw('total_tokens_used + ' . ($in+$out)), 'total_cost_usd' => DB::raw('total_cost_usd + ' . $cost)])` to push the addition into the DB and avoid the read-modify-write race.
- 🔴 **`AiClient::pickProvider` throws `RuntimeException('No active AI provider configured...')` without any fallback or graceful disabling.** When `AiProvider` table is empty (fresh install before admin runs the seeder + adds key) every AI-touching code path 500s. A `BootGuard::aiEnabled()` style helper that returns false when no provider exists would let UI layers degrade gracefully (the way `PaymentController::isEnabled()` does for Midtrans).
- 🟡 **`UsageTracker::PRICING_PER_MTOK` order-matters comment** (line 28-29) is correct, but the table is fragile — adding a `gpt-5-mini-turbo` model later would silently match `gpt-5-mini`'s price. Switch to ordered associative match with explicit regex anchors or a `match (true)` chain.
- 🟡 **`SeoMetaGenerator::parseJson` uses recursive regex `'/\{(?:[^{}]|(?R))*\}/s'`** (line 133) which can hang on pathological inputs. Add a length pre-check (e.g. `mb_strlen($raw) < 8000`) before invoking the recursive matcher.
- 🟡 **`MovieReviewer::parseReview` parses with regex against `TITLE: / RATING: / REVIEW:`** but the multiline `REVIEW:` capture (line 197) is greedy across newlines and will swallow trailing junk (e.g. "Pro tip: try a different perspective for variety!"). Strip everything after the first `\n\n----` or `\n\n===` style fence the models sometimes add.
- 🟡 **`QuoteExtractor::buildNumberedCues` uniform sampling** is a length-only heuristic and can miss dialogue-dense Act 2 climaxes. Sample with a popularity weight (cue density windows from `TrailerSuggester::scoreFromSubtitles` would work) so memorable lines aren't dropped on long films.
- 🟡 **`PromoBannerGenerator::clamp` and `SocialMediaPostGenerator::clamp` append `'…'` (single char)** but the clamp formula does `$max - 1` then appends — works, but the `…` is a multibyte character (3 bytes UTF-8). Length contracts measured in *characters* are fine; if any downstream system counts bytes (some legacy SMS gateways do) this misreports.
- 🟡 **`AdminController::storeAiProvider` validates `'api_key' => 'required|string|min:10|max:255'`** — the minimum-length check is wrong for some providers (Anthropic test keys are 11 chars, fine; Google API keys are 39 chars; but OpenAI session keys for some sandbox modes are 51-56). The 10-char minimum is a friendly nudge but the 255-char `max` truncates pasted OpenAI project keys (`sk-proj-...`) which can exceed 200 chars. Bump to `max:512`.
- 🟡 **`ProviderTester::test` swallows `Http::timeout(15)` cleanly but the `connectTimeout(5)` is on the slow side for global probes** — a Jakarta-hosted box probing Anthropic US-east will spend most of those 5s on TCP/TLS, leaving 10s for the LLM. Bump connectTimeout to 8s.
- 🟡 **`EmailPersonalizer` constructor signature `?AiClient $ai = null`** is documented as "let the container DI null for fallback mode", but Laravel's container *always* resolves typed constructor args via autowiring — you'll always get a real `AiClient`. The fallback path is dead unless someone explicitly does `new EmailPersonalizer(null)`. (Indeed, `AiSendWeeklyDigest.php:46` resorts to `new EmailPersonalizer($ai)` to force the injection.) Either drop the `?` and the fallback paths, or make `$ai` non-nullable + accept a `Closure $aiFactory` for lazier resolution.
- 🟡 **`SocialMediaPostGenerator::FACEBOOK_MAX = 2000`** comment says "engagement drops past ~80 anyway" — if 80 chars is the actual product rule, encode it as a separate `FACEBOOK_RECOMMENDED = 80` constant and warn in the response (e.g. `character_count_warning: 'over recommended'`). Right now the AI can return 1900-char Facebook posts unchallenged.
- 🟡 **`MovieReviewer` writes `provider_used` as `'provider:model'` string** (line 100-103) but no migration adds an index on this column. Filter queries like "show me all reviews generated by Claude" do a full scan.
- 🟢 **`WebSearchService::wikipedia` uses `User-Agent: 'FLiK Assistant/1.0 (https://flik.id)'`** — that domain is asserted but `config('app.name')` is "FLiK" while the production URL might be different. Wikipedia's API policy is to surface a contact email — add `(admin@<config('app.url')>)`.
- 🟢 **`CommentModerator` uses `confidence > 0.7`** (strict greater-than). A 0.70 score is rejected as not-confident-enough. Probably intentional, but worth a test asserting the boundary.

## 🐛 Bugs

- 🔴 **CLAUDE.md documents `comments.moderation_reason`** as a column but the migration creates `moderation_label` + `moderation_score` (no `_reason` column exists — `grep "moderation_reason" database/migrations` → no matches). The CommentModerator does compute a `reason` string and **returns it** from `moderate()` but **never persists it** — once the AI run finishes, the reason is unreachable forever. The moderation queue UI (`resources/views/admin/comments/queue.blade.php`) cannot show "why did the AI flag this comment". Either add a `moderation_reason TEXT NULLABLE` column + persist it, or strip the misleading return value.
- 🔴 **`AiClient::pickProvider` cache hazard.** Because the picked provider is resolved on every `chat()` call against the DB, a Wave-1 sync batch (e.g. `flik:ai:tag-all --sync`) that runs for an hour can pick **different providers across runs** if an admin flips the default mid-batch. Within a single job this is fine (one `chat` call), but multi-call tasks (`CommentSentimentAnalyzer::analyzeBulk` chunks of 40, `MovieReviewer` × 4 perspectives) can mix providers in one logical operation, making `MovieAiReview.provider_used` per-perspective accurate but per-batch confusing. Memoise on the instance.
- 🔴 **`SeoMetaGenerator::buildUserPrompt` uses `$movie->castMembers()->limit(5)->pluck('name')`** when the relation isn't loaded (line 88-90). On a Movie model fetched without `with('castMembers')`, this fires a second query — and `$movie->castMembers()` lazy relation isn't eager-loaded by `AiGenerateSeo::handle` (line 81). N+1 across the catalog when running `flik:ai:seo-all`. Same pattern in `MovieTagger::buildUserPrompt` (line 104-106).
- 🔴 **`DailyAdminReportGenerator::totalWatchHours` sums `progress_seconds` across all watch-history rows** (line 169-173), not the *delta* since yesterday. Result: every rewatch of an old movie inflates "yesterday's hours" by the entire historical progress. Either use `WatchHistory::created_at`-windowed sum, or maintain a `watch_sessions` table that records start/end deltas.
- 🔴 **`DailyAdminReportGenerator::commentsSummary` does keyword sentiment that double-counts.** A comment containing both "bagus" and "boring" hits both branches; the code sets `$hasPos=true; break` then `$hasNeg=true; break` then `elseif ($hasNeg && !$hasPos)` — since `$hasPos=true`, that branch fails, so the comment goes to `neutral` — but the intent of `mixed` is exactly this case. Should call into the real sentiment column or at least add a `mixed` bucket.
- 🔴 **`TrailerSuggester::measureLoudness` uses `'-f', 'null', '-'`** as the FFmpeg null sink — on Windows the trailing `-` is sometimes interpreted as a file path. Use platform-portable `NUL` on Windows / `/dev/null` on POSIX, or feed FFmpeg `nullsink` directly via `-f` `null` `pipe:1` 2>&1.
- 🔴 **`TrailerSuggester::scoreFromAudio` calls `file_put_contents($local, $stream)`** at line 380 — passing a stream resource to `file_put_contents` writes the *result of `stream_get_contents`* (i.e. the whole file) into memory before writing. On a 4 GB feature film this OOMs the worker. Use `stream_copy_to_stream` like `ThumbnailPicker::resolveVideoPath` correctly does (line 340).
- 🔴 **`ThumbnailPicker::pickGeminiProvider` looks up `is_active=true` Gemini rows**, but a switching admin who toggles all `google` providers off has no fallback — task silently returns null without logging *why* to operators. Add a `'No active google provider configured'` warning rather than just `'no Gemini provider configured, skipping'`.
- 🔴 **`AiClient::buildHeaders` for `google` falls through to default**, which then sets `Authorization: Bearer $api_key`. Innocuous because the Google branch in the URL builder appends `?key=...` so the API key is sent twice (header + query), but if Google ever rejects requests with extra Authorization headers (some Vertex AI endpoints do), this breaks. The `switch` should `break;` in the `google` case.

  Code (`AiClient.php:206-219`):
  ```php
  switch ($provider->provider) {
      case 'anthropic':
          $headers['x-api-key'] = $provider->api_key;
          $headers['anthropic-version'] = '2023-06-01';
          break;
      case 'google':
          // API key in URL, no auth header needed
          break;
      default:
          $headers['Authorization'] = 'Bearer ' . $provider->api_key;
  }
  ```
  Actually re-reading — the `break;` IS there, so this is fine. Strike this bullet. (Leaving the note for future audit reuse — the muscle memory was wrong.)
- 🟡 **`MovieReviewer` profile temperature `family=0.55` is colder than `critic=0.6`** while the system prompt asks for a chatty parent-focused review. Sub-optimal; not a bug.
- 🟡 **`CommentSentimentAnalyzer::analyzeBulk` returns void**, and the inner `analyzeChunk` continues on chunk failure with only a `Log::warning` — caller has no way to know if 2 of 40 chunks failed silently. Return a per-comment result array so the queue layer can fail the job on bad ratios.
- 🟡 **`QuoteExtractor::fallbackFromOverview` regex `'/[“"]([^“”"]{8,200})[”"]/u'`** only matches straight + smart double-quotes, ignoring single quotes. Indonesian dialog often uses single quotes inside narration. Add `'\'‘’'` alternatives.
- 🟡 **`TrailerSuggester::EMOTION_KEYWORDS`** has English `'kill'` but not Indonesian `'bunuh'` in word-boundary form — both are in the list but the `substr_count(' '.$lower.' ', ' '.$kw.' ')` boundary trick falls apart for Indonesian agglutination (`membunuh`, `dibunuh`) which won't match `bunuh`. Strip suffixes or relax to `str_contains`.
- 🟡 **`AdminController::aiSettings` returns `view('admin.ai-settings', ...)`** (line 544) — note the dash; the view path is `resources/views/admin/ai-settings.blade.php`. The convention elsewhere in the admin namespace is dot-notation (`admin.users.index`). Works because the file exists, but inconsistency surprises new contributors.
- 🟡 **`UsageTracker::FALLBACK_PRICE_PER_MTOK = [1.00, 3.00]`** is conservative for cheap models (DeepSeek V4 Flash is $0.14/$0.28) but undercharges expensive ones (Claude Opus is $15/$75). A typo in the model name on `AiProvider` row silently understates cost by 75×. Add a `Log::warning('UsageTracker: unknown model pricing fallback', ['model' => $modelName])` when `resolvePricing` falls through.

## Open questions

- **Should the moderation `is_visible=false` apply retroactively to ALREADY-displayed comments?** Right now the bug above means flagged comments stay visible, so the question hasn't surfaced. Once that fix lands, a comment that's been visible for an hour might suddenly disappear when moderation completes — UX-jarring. Consider a soft "Under review" placeholder instead of full removal.
- **What's the intended cadence for `CommentSentimentAnalyzer`?** Single-call per new comment via listener (consistent with moderation+spoiler) or nightly bulk (the analyzer is *designed* for bulk and chunks at 40 to amortise cost)? Pick one — currently neither is wired.
- **Is `flik:ai:reviews-all` a missing command, or is the per-perspective UI workflow the design intent?** Generating 4 perspectives × 350 movies = 1400 LLM calls; without a CLI it's labour-impossible.
- **Why does `AiClient` log warnings on non-2xx but `RuntimeException` on parse failure?** Half the call sites swallow `\Throwable`, half let it bubble. Document the contract (e.g. "always throws on failure; caller MUST try/catch") and remove the half-and-half.
- **Should there be a hard cost ceiling per provider per day** (e.g. `$AiProvider->daily_budget_usd`) that `AiClient::pickProvider` honours by skipping over-budget providers and surfacing on `/admin/ai-usage`? Today a runaway loop can quietly burn the budget for the month.
- **`provider_used` on `MovieAiReview` is a string** — should it become a foreign key to `ai_providers.id` (with `nullable` for deleted provider rows)? Currently if an admin deletes a provider, the historical review still references the dead provider's label string but can't be filtered by `is_active` etc.
- **`thumbnail-tmp` and `trailer-tmp` directories under `storage/app/`** are never cleaned. Long-running install accumulates GBs of keyframe JPEGs + downloaded MP4 copies. Add a `flik:cleanup:tmp` command and a weekly schedule.
